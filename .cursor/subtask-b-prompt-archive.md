## 测试与性能评估规则（Trae subtask-b 技能补充）

### BlessStar 测试接入约定（CMake + CTest + 脚本）

与本仓库当前工程一致时，按下列方式接入；**单元/边界/综合可执行测试**与 **性能基准（benchmark）** 分流处理。

#### 目录与职责

| 内容 | 路径或目标 | 说明 |
|------|------------|------|
| 测试源码 | 各模块下 `kernel/**/test/`、`adapter/test/` 等 | **与模块同仓**，不集中到单一 `tests/` 根目录 |
| 测试目标定义与 CTest 注册 | `cmake/Tests.cmake` | 根 `CMakeLists.txt` 在库目标就绪后 `include(cmake/Tests.cmake)`；此处统一 `add_executable`、`target_link_libraries`、`add_test` |
| 聚合运行 | CMake 自定义目标 `run_all_tests` | 内部调用 `ctest`，并依赖已注册的单元测试可执行目标，便于「先编后测」 |
| 本地脚本（Windows） | `tools/test/run.ps1` | 封装 `ctest`：`BuildDir`（默认 `build`）、`Config`（多配置生成器必填时可传；否则可用环境变量 `BLESSSTAR_CTEST_CONFIG`） |
| 本地脚本（Windows，仅 C/C++） | `tools/dev/format_cpp_sources.ps1` | 只对 `kernel`/`adapter`/`tests`/`cmake` 下 `.cpp/.h/...` 调 `clang-format -i`；**禁止**对 `tools/**/*.py` 误跑 clang-format |
| CI | `.github/workflows/ci.yml` | Unix 单配置 + Windows 多配置下对 `ctest` 的调用方式已区分，落地后应保持与本地脚本行为一致 |

#### 新增单元测试时的 CMake 接入步骤

1. 在对应模块的 `test/` 目录新增 `*.cpp`（命名与现有风格一致，可执行目标名建议 `bs_test_*`）。  
2. 在 **`cmake/Tests.cmake`** 中增加一次 **`blessstar_add_unit_test`** 调用，例如：  
   - `SOURCES`：一个或多个源文件路径（相对仓库根）。  
   - `LIBS`：该测试依赖的 CMake 目标（空格分隔多个库）。  
3. 若测试依赖的**库目标**尚不存在：先在根 **`CMakeLists.txt`** 中 `add_library` / `target_link_*`，并保证其出现在 **`include(cmake/Tests.cmake)` 之前**。不要在 `Tests.cmake` 中定义库。  
4. 需要调整 CTest 属性（如更慢的综合测）：在对应 `blessstar_add_unit_test(...)` **之后**用 `set_tests_properties(...)` 追加 `LABELS`、`TIMEOUT` 等（`blessstar_add_unit_test` 内已为单元测设置默认标签与超时）。  
5. **性能基准可执行文件**（`bs_benchmark_*`）：仅在根 **`CMakeLists.txt`** 中 `add_executable` + 链接，**不**调用 `blessstar_add_unit_test`，即**不注册到 CTest**，避免与单元回归混跑。  
6. **变更台账**：上述文件变更落地后，按规则写入 **`项目修改记录.md`**（含 `cmake/Tests.cmake`、新增 `*.cpp` 等路径）。

#### `blessstar_add_unit_test` 示例（`cmake/Tests.cmake`）

```cmake
blessstar_add_unit_test(bs_test_mymodule_foo
  SOURCES kernel/mymodule/test/FooTest.cpp
  LIBS bs_kernel_mymodule
)

# 多库依赖示例
blessstar_add_unit_test(bs_test_mymodule_bar
  SOURCES kernel/mymodule/test/BarTest.cpp
  LIBS bs_kernel_pipeline bs_kernel_report
)
```

#### 子任务B 执行回归时的推荐命令

- **Windows（PowerShell）**：`.\tools\test\run.ps1 -BuildDir <构建目录> [-Config Release]`  
- **Linux / macOS / Git Bash**：`./tools/test/run.sh <构建目录> [Release]`（若不可执行先 `chmod +x tools/test/run.sh`）  
- **通用 / CI 对齐**：在构建目录执行  
  `ctest --test-dir <构建目录> --output-on-failure`  
  （Visual Studio 等多配置生成器需加 `-C <配置>`，与 `cmake --build` 所用配置一致。）  
- **同 CMake 一键**：`cmake --build <构建目录> --target run_all_tests`（多配置下仍需保证已用正确配置完成编译，必要时对底层 `ctest` 使用 `-C`）。  

若维护 **Meson** 侧一致性：在 `meson.build` 中为同一测试增加 `executable` + `test()`，源路径与 CMake 侧保持一致。

### 测试执行流程

1. **测试设计**：以当日子任务A/主任务确认的功能与设计依据为准。  
2. **测试代码编写**：覆盖边界条件、正常流程、异常情况；**按「BlessStar 测试接入约定」写入模块 `test/` 并在 `cmake/Tests.cmake` 注册（benchmark 除外）**。

4. **问题修复**：测试失败时定位并修改源代码，直到测试通过（以日程/决策为准）。  
5. **测试代码清理**：若存在临时测试代码且流程要求清理，则在用户确认当日代码达标后按约定删除或归档。
